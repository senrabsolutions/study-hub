# AI-Assisted CLI Workflows

## Why this matters

Claude Code is an AI tool that operates inside developer CLI workflows. Understanding how AI-assisted workflows actually function -- not just "AI helps you code" but the specific mechanics of how Claude Code integrates with your shell, your tools, and your context -- is foundational to this role.

This lesson covers what good AI-CLI integration looks like, what the failure modes are, and how to think about developer trust and safety in AI-powered workflows.

---

## Key Concepts

- AI-assisted workflows augment the developer -- they do not replace judgment
- Shell context is critical: the AI must know which shell is active, what PATH contains, and what the current environment looks like
- Tool invocations are the mechanism: the AI requests actions, the tool executes them and reports results
- Validation before execution is non-negotiable for destructive operations
- The feedback loop (execute -> observe -> adjust) is what makes agentic workflows powerful

---

## How Claude Code integrates with the shell

When Claude Code runs, it has a context about the developer's environment:

```
Shell type:          PowerShell 7 (pwsh)
Working directory:   C:\Users\logan\project
OS:                  Windows 11
Node version:        20.11.0
Git branch:          feature/windows-fix
```

This context shapes how Claude generates commands. A well-configured integration means Claude generates PowerShell syntax for PowerShell users, CMD syntax for CMD users, and bash for WSL users.

**When context is wrong, commands fail:**

```powershell
# Claude thinks you are using bash (wrong context)
# Generates: find . -name "*.ts" | xargs grep "import"
# User is in CMD -- this command does not exist

# Claude knows you are using PowerShell (correct context)
# Generates: Get-ChildItem -Recurse -Filter "*.ts" | Select-String "import"
# Works correctly
```

The Windows platform engineer's job includes ensuring Claude Code correctly detects and uses the shell context.

## The validation principle

AI-generated commands must be treated as suggestions, not instructions. This is especially important for:

**Destructive operations:**
```powershell
# A command Claude might generate to clean up node_modules:
Remove-Item -Recurse -Force .\node_modules

# Before running this, verify:
# 1. Is this the right directory?
# 2. Is this the right project?
# 3. Will this break anything currently running?
```

**Commands that assume your environment:**
```bash
# AI generates this (assuming Unix):
npm config set prefix ~/.npm-global

# On Windows this should be:
npm config set prefix "$env:APPDATA\npm"

# The AI may have generated an incorrect path for your platform
```

**Commands with side effects beyond the immediate task:**
```powershell
# AI generates this to fix a PATH issue:
[Environment]::SetEnvironmentVariable("PATH", "$env:PATH;C:\new\path", "Machine")
# This permanently modifies the machine-level PATH for ALL users
# Should it be User scope instead?
```

The principle: understand what a command will do before running it, not after.

## Verifying AI-generated commands before execution

Practical verification steps:

### For file operations:

```powershell
# Before running a destructive command, preview what it would affect
Get-ChildItem -Recurse -Filter "*.tmp" | Select-Object FullName
# Then decide if Remove-Item on these is what you want
```

### For shell commands:

```powershell
# Check if a command exists before running something that depends on it
Get-Command git -ErrorAction SilentlyContinue
Get-Command node -ErrorAction SilentlyContinue

# Check what version would run
(Get-Command node).Source
node --version
```

### For environment changes:

```powershell
# Before modifying PATH, check current state
$env:PATH -split ";"

# Understand scope: User vs Machine
[Environment]::GetEnvironmentVariable("PATH", "User")
[Environment]::GetEnvironmentVariable("PATH", "Machine")
```

## Shell context errors -- the most common failure mode

The most frequent AI-CLI failure is context mismatch: the AI generates a command for the wrong shell.

```
AI thinks: bash
User has:  PowerShell
Generated: ls -la | grep "*.ts"
Actual:    PowerShell does not have ls with Unix flags; grep is not available
```

**How to detect context errors:**

The error message often reveals the mismatch:
- `'grep' is not recognized` -- command is from Unix, you are in CMD/PowerShell
- `The term 'Get-ChildItem' is not recognized` -- PowerShell command in CMD
- `bash: syntax error near unexpected token` -- Windows syntax in bash

**How to fix them:**

```powershell
# Tell Claude Code your context explicitly
# In many tools, you can provide this in the prompt:
"I am using PowerShell 7 on Windows 11. Please use PowerShell syntax."

# Or verify the generated command before running:
# - Does it use %VAR% (CMD) or $env:VAR (PowerShell)?
# - Does it use backslashes or forward slashes in paths?
# - Does it use Unix tools (grep, awk, sed) or PowerShell equivalents?
```

## The agentic feedback loop

What makes Claude Code powerful is not individual commands -- it is the ability to execute, observe results, and adjust:

```
User: "Fix the failing tests"

Claude Code:
1. Run tests -> see which tests fail
2. Read the failing test files
3. Read the implementation files they test
4. Generate a fix
5. Apply the fix
6. Run tests again -> verify they pass
7. Report result
```

Each step's output feeds into the next. This feedback loop can accomplish complex tasks without constant user input.

**Where it goes wrong on Windows:**

- Test runner output format differs between shells -- parsing may fail
- File paths in error output may use different separators
- Exit codes may differ between Windows and Unix tools
- stdout/stderr mixing can confuse output parsing

Each of these is a Windows-specific integration point that the platform engineer owns.

## Safety patterns in AI-CLI workflows

Well-designed AI-CLI tools implement several safety patterns:

**Show before execute:**
```
Claude Code: I want to run the following command:
  Remove-Item -Recurse .\dist

Proceed? [y/N]
```

**Dry run mode:**
```
Claude Code: In dry run mode. Would have:
  - Deleted: .\dist\index.js
  - Created: .\dist\index.js (new version)
  - Ran: npm test
```

**Scope limitation:**
Claude Code only operates within the current working directory by default. It should not read or write files outside the project directory without explicit confirmation.

**Undo capability:**
For file modifications, keeping a git-staged or backed-up version allows reverting AI changes if they are wrong.

## What to Notice

- Shell context detection is an active engineering responsibility -- if Claude generates wrong-shell commands, the integration is broken
- Verification before execution is not optional for destructive commands -- the AI has context but not certainty
- The agentic feedback loop is powerful but requires correct output parsing at each step -- Windows output formats differ from Unix
- Safety patterns (show before execute, scope limitation, undo) are engineering decisions that directly affect developer trust
