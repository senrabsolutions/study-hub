# Versioning & Update Strategy for CLI Tools

## Why this matters

Building a CLI tool is only half the problem — the other half is:

- how it gets updated
- how breaking changes are handled
- how users stay on compatible versions

Poor versioning leads to:

- broken workflows
- frustrated users
- hard-to-debug issues

Good versioning builds **trust and reliability**.

---

## Key Concepts

### Semantic Versioning (SemVer)

Most CLI tools follow:

```
MAJOR.MINOR.PATCH
```

- **MAJOR** → breaking changes  
- **MINOR** → new features (backward compatible)  
- **PATCH** → bug fixes  

Example:

```
1.2.3 → 2.0.0 (breaking change)
```

---

### Breaking Changes

A breaking change is anything that:

- changes CLI behavior
- modifies command structure
- removes features
- alters output format

---

### Update Channels

Tools may support:

- stable releases
- beta versions
- nightly builds

---

### Update Mechanisms

Tools are typically updated via:

- package managers (winget, choco, scoop)
- npm (`npm install -g`)
- built-in self-update commands

---

## Quick Summary

Versioning defines how changes are communicated to users. A good update strategy ensures stability while allowing the tool to evolve.

---

## Quick Lab

### Step 1 — Check version of a tool

```bash
node --version
```

### Step 2 — Simulate version change

```json
{
  "version": "1.0.0"
}
```

Update to:

```json
{
  "version": "2.0.0"
}
```

---

## What to Notice

- Version numbers communicate risk
- Major changes signal breaking updates
- Users rely on predictable versioning
- Poor versioning leads to confusion

---

## Real-World Example

A CLI tool updates from:

```
1.5.0 → 2.0.0
```

Suddenly:

- commands behave differently
- scripts break

Why?

Because a **breaking change** was introduced.

---

## Debugging Strategy

When a tool suddenly behaves differently:

1. Check the version:

```bash
tool --version
```

2. Look for recent updates
3. Review changelog
4. Downgrade if necessary

---

## Interview Angle

You should be able to explain:

- Semantic versioning communicates change impact
- Breaking changes require major version bumps
- Update strategy affects user trust
- Stability is critical for developer tools

Strong answer example:

“I would follow semantic versioning so users understand the impact of updates. Breaking changes go into major versions, while minor and patch updates maintain compatibility. This helps prevent unexpected breakage and builds trust.”
