# winget: Windows Package Manager

## Why this matters

winget is Microsoft's official package manager, built into Windows 10/11. For a Windows platform engineer, understanding winget deeply matters for two reasons: distributing CLI tools to Windows users cleanly, and diagnosing installation failures when winget is part of the problem.

Claude Code needs reliable installation on Windows. winget is one of the primary distribution channels. Knowing how winget works under the hood helps you design better installers and debug the failures users report.

---

## Key Concepts

- winget is built into Windows 10 (1809+) and Windows 11 via the App Installer package
- Packages come from the WinGet Community Repository on GitHub
- winget handles installation, upgrade, uninstall, and search
- Package manifests define how winget finds, downloads, installs, and verifies packages
- winget does not always update PATH automatically -- this is a common source of "installed but not found" reports

---

## Basic usage

```powershell
# Search for a package
winget search nodejs
winget search "Visual Studio Code"

# Install a package
winget install OpenJS.NodeJS
winget install Microsoft.VisualStudioCode
winget install Git.Git

# Install silently (no prompts -- useful for automation)
winget install OpenJS.NodeJS --silent

# Install a specific version
winget install OpenJS.NodeJS --version 20.11.0

# Show installed packages
winget list

# Check for upgrades
winget upgrade

# Upgrade a specific package
winget upgrade OpenJS.NodeJS

# Upgrade everything
winget upgrade --all

# Uninstall
winget uninstall OpenJS.NodeJS

# Show package details
winget show OpenJS.NodeJS
```

---

## Package IDs

Every winget package has a unique ID in the format `Publisher.PackageName`:

```
OpenJS.NodeJS
Microsoft.VisualStudioCode
Git.Git
Anthropic.Claude
JetBrains.WebStorm
```

Using the full ID is more reliable than searching by name -- names can be ambiguous and match multiple packages.

```powershell
# Ambiguous -- might match multiple packages
winget install node

# Explicit -- no ambiguity
winget install OpenJS.NodeJS
```

---

## How winget installs packages

winget is a package manager, not an installer itself. It:

1. Looks up the package manifest in the WinGet Community Repository
2. Downloads the installer (MSI, MSIX, EXE, ZIP, etc.)
3. Runs the installer silently or interactively
4. Optionally verifies the hash of the downloaded file

This means winget's install experience is only as good as the underlying installer. If an MSI installs to a non-standard location and does not add itself to PATH, winget cannot fix that.

## PATH and winget

This is the most common issue users report after using winget:

```powershell
winget install OpenJS.NodeJS
# ✓ Successfully installed

node --version
# 'node' is not recognized as an internal or external command
```

What happened: Node was installed but the current PowerShell session does not have the updated PATH. The installer added Node to PATH, but the current session inherited the old PATH before the install.

**Fixes:**

```powershell
# Option 1: Open a new terminal (picks up updated PATH)
# Close PowerShell, open a new one

# Option 2: Refresh PATH in current session
$env:PATH = [System.Environment]::GetEnvironmentVariable("PATH", "Machine") + ";" + 
            [System.Environment]::GetEnvironmentVariable("PATH", "User")

# Option 3: Use the full path temporarily
& "C:\Program Files\nodejs\node.exe" --version
```

## winget source management

winget pulls packages from configured sources. The default source is the WinGet Community Repository:

```powershell
# List configured sources
winget source list

# Output:
# Name    Argument
# ------  -----------------------------------------
# winget  https://cdn.winget.microsoft.com/cache

# Add a custom source (for enterprise/internal packages)
winget source add --name mycompany --arg https://internal.example.com/winget

# Update source cache
winget source update
```

Enterprise environments often configure a custom winget source pointing to an internal package feed. This lets them distribute internally-approved tools the same way users install public packages.

---

## Exporting and restoring packages

winget can export your installed packages and restore them on a new machine -- useful for dev environment setup:

```powershell
# Export all installed packages to a JSON file
winget export -o packages.json

# Import and install everything from that file
winget import -i packages.json
```

This is similar to a `requirements.txt` for your Windows dev environment.

## The winget manifest format

Understanding manifests helps you publish and debug packages. A minimal winget manifest:

```yaml
PackageIdentifier: Anthropic.ClaudeCode
PackageVersion: 1.0.0
PackageName: Claude Code
Publisher: Anthropic
License: Proprietary
ShortDescription: AI-powered coding assistant CLI
Installers:
  - Architecture: x64
    InstallerType: exe
    InstallerUrl: https://releases.anthropic.com/claude-code/1.0.0/claude-code-win-x64.exe
    InstallerSha256: <sha256 hash>
    InstallerSwitches:
      Silent: /S
      SilentWithProgress: /S
ManifestType: singleton
ManifestVersion: 1.4.0
```

The `InstallerSha256` is verified after download -- if the hash does not match, winget refuses to run the installer. This is the security verification step.

## Comparing winget to Chocolatey

| Feature | winget | Chocolatey |
|---|---|---|
| Built into Windows | Yes (Win 10 1809+) | No -- must install |
| Package count | ~6,000+ | ~9,000+ |
| Enterprise features | Limited | Extensive (paid) |
| Custom feeds | Yes | Yes |
| Script execution | No | Yes (PowerShell install scripts) |
| Reliability | Improving | Battle-tested |

For consumer/developer tools, winget is usually fine. For enterprise deployment with complex installation logic, Chocolatey's PowerShell scripts offer more control.

## Automating with winget in scripts

```powershell
# Check if a package is installed
$installed = winget list --id OpenJS.NodeJS 2>$null
if ($LASTEXITCODE -eq 0) {
    Write-Host "Node.js is installed"
} else {
    Write-Host "Node.js is not installed"
    winget install OpenJS.NodeJS --silent --accept-package-agreements --accept-source-agreements
}
```

Key flags for scripting:
- `--silent` -- no interactive prompts
- `--accept-package-agreements` -- accept license agreements automatically
- `--accept-source-agreements` -- accept source terms automatically
- `--disable-interactivity` -- fail rather than prompt

## What to Notice

- The full package ID (`Publisher.PackageName`) is more reliable than searching by name
- PATH is not automatically refreshed in the current session after winget installs something -- this causes most "installed but not found" reports
- winget is a front-end -- the actual install is done by MSI, EXE, or MSIX installers; their quality varies
- `--accept-package-agreements` and `--accept-source-agreements` are required for fully non-interactive scripting
