# Subprocess Spawning

## Why this matters

A huge percentage of CLI tools don’t do everything themselves -- they **call other tools**.

Examples:
- a Node CLI calling `git`
- a script invoking `npm`
- a build tool orchestrating multiple commands

When this breaks, you get issues like:

> “This works in my terminal, but fails inside the tool.”

Understanding subprocess spawning is critical for debugging these failures.

---

## Key Concepts

- A **subprocess** is a child process created by another process
- It inherits:
  - environment variables (PATH, etc.)
  - working directory
  - permissions

- In Node.js, common APIs:
```js
require("child_process").exec
require("child_process").spawn
require("child_process").execFile
```

---

## Quick Summary

Subprocesses run commands from within programs. Their behavior depends entirely on the parent process environment and execution context.

---

## Real-World Scenario

A CLI tool runs:

```bash
git clone repo-url
```

Inside the tool, it fails with:

```
git: command not found
```

But running the same command in your terminal works.

---

## Why This Happens

- Your terminal has Git in PATH  
- The parent process (the CLI tool) does not  
- The subprocess inherits the parent’s environment  

Result:
👉 the subprocess cannot locate `git`

---

## Quick Lab

### Step 1 -- Run a subprocess in Node

```bash
node -e "require('child_process').exec('echo hello', (e,o)=>console.log(o))"
```

---

### Step 2 -- Test PATH-dependent command

```bash
node -e "require('child_process').exec('git --version', (e,o,err)=>console.log(o||err))"
```

---

### Step 3 -- Inspect environment

```bash
node -e "console.log(process.env.PATH)"
```

---

## What to Notice

- subprocess behavior depends on PATH  
- failures often come from missing environment variables  
- stdout and stderr must be handled correctly  
- tools may fail silently if errors are not captured  

---

## Deep Dive (Important)

Subprocess execution chain:

```text
Terminal → CLI Tool → Subprocess
```

Each level inherits from the previous one.

If something is wrong upstream, everything downstream breaks.

---

## Real-World Example

A tool works in terminal but fails in VS Code.

Why?

- VS Code was launched before PATH was updated  
- CLI tool inherits VS Code’s environment  
- subprocess inherits CLI tool’s environment  

Result:
👉 command fails even though it is installed

---

## Debugging Strategy

When a subprocess fails:

### Step 1 -- Reproduce manually
Run the same command in the terminal

---

### Step 2 -- Check PATH

```js
console.log(process.env.PATH)
```

---

### Step 3 -- Capture stderr

```js
exec("git --version", (err, stdout, stderr) => {
  console.log(stderr);
});
```

---

### Step 4 -- Check working directory

```js
console.log(process.cwd());
```

---

### Step 5 -- Compare environments
- terminal vs script  
- CMD vs PowerShell  
- IDE vs standalone terminal  

---

## Interview Angle

You should be able to explain:

- Subprocesses inherit environment from the parent  
- PATH issues are a common cause of failure  
- stdout/stderr handling is critical  
- debugging requires comparing environments  

Strong answer:

“When a subprocess fails, I check the parent process environment, especially PATH, and compare it to a working terminal. Since subprocesses inherit their environment, differences there often explain the issue.”
