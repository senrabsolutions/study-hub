# Config Precedence

## Why this matters

Every serious CLI tool has multiple ways to configure behavior:

- command-line flags
- environment variables
- config files
- defaults

If you don’t define clear precedence, your tool becomes unpredictable.

Understanding config precedence helps you:

- debug why a tool is behaving a certain way
- design tools that are intuitive and reliable
- explain real-world CLI behavior in interviews

This is one of the most common patterns in modern tooling.

---

## Key Concepts

- CLI tools often support multiple configuration sources:
  - command-line arguments (`--port=3000`)
  - environment variables (`PORT=3000`)
  - config files (`config.json`)
  - default values
- These sources must follow a **defined priority order**
- The most common precedence model:

```text
CLI flags > Environment variables > Config files > Defaults
```

- Higher-precedence values override lower ones

---

## Quick Summary

Config precedence determines which value wins when multiple configuration sources define the same setting.

---

## Quick Lab

### Step 1 — Simulate environment variable

```powershell
$env:PORT=4000
```

### Step 2 — Simulate CLI argument

```bash
node app.js --port=3000
```

### Step 3 — Inside your app (conceptually)

```js
const port =
  cliArgPort ||
  process.env.PORT ||
  configFile.port ||
  8080;
```

---

## What to Notice

- CLI arguments override everything else
- Environment variables override config files
- Defaults are only used if nothing else is provided
- This creates predictable, debuggable behavior

---

## Real-World Example

A CLI tool runs with:

- config file: `port = 5000`
- environment variable: `PORT=4000`
- CLI flag: `--port=3000`

Result:

```
3000
```

Why?

Because CLI flags have the highest precedence.

---

## Debugging Strategy

When a tool behaves unexpectedly:

1. Check CLI arguments first
2. Check environment variables
3. Check config files
4. Confirm default values

Always think:

👉 “Which layer is overriding which?”

---

## Interview Angle

You should be able to explain:

- Tools support multiple configuration sources
- A clear precedence order prevents confusion
- CLI flags typically override everything
- This pattern is used across many real-world tools

Strong answer example:

“I would define a clear precedence order where CLI arguments override environment variables, which override config files, which override defaults. That ensures users can always explicitly control behavior when needed.”
