# Config Precedence in CLI Tools

## Why this matters

Claude Code resolves configuration from multiple sources -- environment variables, config files, command-line flags, and defaults. When a user reports "my setting is being ignored" or "the wrong API key is being used," the answer is almost always a precedence issue: one source is overriding another in a way the user did not expect.

Understanding how precedence works -- and how to implement it correctly -- is essential for debugging these issues and for building CLI tools that behave predictably.

---

## Key Concepts

- CLI tools typically support four configuration sources in decreasing priority: flags > env vars > config files > defaults
- Higher-priority sources override lower-priority sources for the same setting
- Users expect explicit overrides (flags) to win over implicit configuration (config files)
- Merging vs overriding matters for object/array settings
- Config file location resolution has its own precedence (project-level vs user-level vs system-level)

---

## The standard precedence model

```
1. Command-line flags         --api-key=sk-...        (highest priority)
2. Environment variables      ANTHROPIC_API_KEY=sk-... 
3. Project config file        .clauderc or claude.json in project root
4. User config file           ~/.claude/config.json
5. System config file         /etc/claude/config.json
6. Default values             hardcoded in the tool     (lowest priority)
```

This ordering matches user expectations: explicit flags override everything, environment variables override config files (useful for CI), and config files override defaults.

## Implementing precedence in TypeScript

```typescript
interface Config {
  apiKey: string;
  model: string;
  maxTokens: number;
  debug: boolean;
}

async function loadConfig(argv: string[]): Promise<Config> {
  // Load each source independently
  const defaults = loadDefaults();
  const systemConfig = await loadConfigFile("/etc/claude/config.json");
  const userConfig = await loadConfigFile(path.join(os.homedir(), ".claude", "config.json"));
  const projectConfig = await loadConfigFile(path.join(process.cwd(), ".clauderc"));
  const envConfig = loadFromEnv();
  const flagConfig = parseFlags(argv);

  // Merge in priority order -- later sources override earlier ones
  return {
    ...defaults,
    ...systemConfig,
    ...userConfig,
    ...projectConfig,
    ...envConfig,
    ...flagConfig,
  };
}
```

The spread operator handles precedence cleanly: each source overwrites keys from lower-priority sources. Later spread wins.

## Loading from environment variables

```typescript
function loadFromEnv(): Partial<Config> {
  const config: Partial<Config> = {};

  // Map env var names to config keys
  const envMapping: Record<string, keyof Config> = {
    ANTHROPIC_API_KEY: "apiKey",
    CLAUDE_MODEL: "model",
    CLAUDE_MAX_TOKENS: "maxTokens",
    CLAUDE_DEBUG: "debug",
  };

  for (const [envVar, configKey] of Object.entries(envMapping)) {
    const value = process.env[envVar];
    if (value !== undefined) {
      // Type coercion based on expected type
      if (configKey === "maxTokens") {
        config[configKey] = parseInt(value, 10);
      } else if (configKey === "debug") {
        config[configKey] = value === "1" || value.toLowerCase() === "true";
      } else {
        (config as Record<string, unknown>)[configKey] = value;
      }
    }
  }

  return config;
}
```

## Config file location resolution

Where does the tool look for config files? This itself has a precedence:

```typescript
async function findConfigFile(filename: string): Promise<string | null> {
  // 1. Check explicit path from env var
  const explicit = process.env.CLAUDE_CONFIG;
  if (explicit && await fileExists(explicit)) return explicit;

  // 2. Walk up from current directory to find project-level config
  let dir = process.cwd();
  while (true) {
    const candidate = path.join(dir, filename);
    if (await fileExists(candidate)) return candidate;
    
    const parent = path.dirname(dir);
    if (parent === dir) break;  // reached root
    dir = parent;
  }

  // 3. User-level config
  const userConfig = path.join(os.homedir(), ".claude", filename);
  if (await fileExists(userConfig)) return userConfig;

  // 4. System-level config
  const systemConfig = process.platform === "win32"
    ? path.join(process.env.ProgramData ?? "C:\\ProgramData", "claude", filename)
    : `/etc/claude/${filename}`;
  if (await fileExists(systemConfig)) return systemConfig;

  return null;
}
```

## The Windows config location

On Windows, config files should follow the platform conventions:

```typescript
function getConfigDir(): string {
  if (process.platform === "win32") {
    // Use APPDATA for user config -- never C:\Program Files
    return path.join(process.env.APPDATA ?? os.homedir(), "Claude");
  } else if (process.platform === "darwin") {
    // macOS: ~/Library/Application Support/ or ~/.config/
    return path.join(os.homedir(), ".config", "claude");
  } else {
    // Linux: XDG Base Directory spec
    return path.join(process.env.XDG_CONFIG_HOME ?? path.join(os.homedir(), ".config"), "claude");
  }
}
```

This is a concrete Windows platform difference. Tools that hardcode `~/.config/tool` work on macOS and Linux but put config in an unusual location on Windows.

## Debugging precedence issues

When a user says "my setting is being ignored":

```typescript
// Add a debug mode that shows where each config value came from
async function loadConfigWithTrace(argv: string[]): Promise<Config> {
  const sources = {
    defaults: loadDefaults(),
    system: await loadConfigFile(systemConfigPath),
    user: await loadConfigFile(userConfigPath),
    project: await loadConfigFile(projectConfigPath),
    env: loadFromEnv(),
    flags: parseFlags(argv),
  };

  if (process.env.CLAUDE_DEBUG_CONFIG) {
    for (const [sourceName, sourceConfig] of Object.entries(sources)) {
      for (const [key, value] of Object.entries(sourceConfig ?? {})) {
        if (value !== undefined) {
          console.error(`[config] ${key} = ${JSON.stringify(value)} (from ${sourceName})`);
        }
      }
    }
  }

  return { ...sources.defaults, ...sources.system, ...sources.user, 
           ...sources.project, ...sources.env, ...sources.flags };
}
```

This is invaluable for support. A user can run with `CLAUDE_DEBUG_CONFIG=1` and immediately see which source is winning for each setting.

## Merging vs overriding

Simple values (strings, numbers, booleans) are straightforward to override. Objects and arrays need a decision:

```typescript
// Simple override -- later source replaces entire object
const merged = { ...defaults.proxy, ...userConfig.proxy };

// Deep merge -- later source only replaces specified keys
function deepMerge<T>(base: T, override: Partial<T>): T {
  const result = { ...base };
  for (const [key, value] of Object.entries(override)) {
    if (value !== null && typeof value === "object" && !Array.isArray(value) &&
        key in result && typeof result[key as keyof T] === "object") {
      result[key as keyof T] = deepMerge(result[key as keyof T], value as Partial<T[keyof T]>);
    } else {
      result[key as keyof T] = value as T[keyof T];
    }
  }
  return result;
}
```

The right choice depends on user expectations. For a list of allowed commands, deep merge is usually wrong -- users expect their list to replace the default, not extend it.

## What to Notice

- Flags beat everything -- a user with a flag override should always win
- Environment variables are the right mechanism for CI configuration -- they override config files without modifying them
- Config file walking (searching parent directories) is how project-level vs user-level config works -- the nearest config to the current directory wins
- On Windows, config should go in `%APPDATA%` not `~/.config` -- `~` on Windows resolves to a different location than Unix engineers expect
