# Terminal I/O: stdin, stdout, stderr, and TTY Detection

## Why this matters

Claude Code is a CLI tool that reads input, writes output, and needs to behave correctly whether it is running interactively in a terminal, piped into another tool, or running in a CI environment with no terminal at all. Getting terminal I/O wrong causes garbled output, broken pipes, and tools that behave differently in automation than in interactive use.

This is one of the areas where Windows behaves differently from Unix in ways that matter.

---

## Key Concepts

- Every process has three standard streams: stdin (0), stdout (1), stderr (2)
- Streams can be redirected or piped -- the program does not always know where they go
- TTY detection determines whether the output is a real terminal or a pipe/file
- Color codes and progress indicators should only be written to a real terminal
- Windows has different TTY detection APIs than Unix -- and some terminal emulators misreport

---

## The three streams

```
stdin  (fd 0) -- data coming IN to the program
stdout (fd 1) -- normal output going OUT
stderr (fd 2) -- errors and diagnostics going OUT
```

```javascript
// Node.js
process.stdin   // Readable stream
process.stdout  // Writable stream
process.stderr  // Writable stream

// Writing to stdout vs stderr
process.stdout.write("normal output\n");
process.stderr.write("error message\n");
console.log("goes to stdout");
console.error("goes to stderr");
```

**Why the distinction matters:**

```powershell
# Redirect stdout to a file -- stderr still goes to terminal
node app.js > output.txt

# Redirect stderr to a file -- stdout still goes to terminal
node app.js 2> errors.txt

# Redirect both
node app.js > output.txt 2>&1

# Pipe stdout to another command -- stderr bypasses the pipe
node app.js | another-tool
```

If a tool writes error messages to stdout instead of stderr, they get captured by pipes and break downstream tools that expect only data on stdout.

---

## TTY detection

A TTY (teletypewriter) is a real interactive terminal. When stdout is a TTY, it makes sense to:
- Output ANSI color codes
- Show progress spinners
- Use interactive prompts
- Format output for human reading

When stdout is a pipe or file, you should:
- Output plain text without color codes
- Skip progress indicators
- Never use interactive prompts (they hang)
- Output machine-parseable format if possible

```javascript
// Node.js TTY detection
const isInteractive = process.stdout.isTTY;   // true if stdout is a terminal
const stdinIsTTY = process.stdin.isTTY;        // true if stdin is a terminal

if (isInteractive) {
  // Safe to use colors and progress indicators
  console.log("\x1b[32mSuccess\x1b[0m");  // green text
} else {
  // Plain output for pipes and CI
  console.log("Success");
}
```

## Windows TTY detection complications

On Windows, TTY detection is more complex because of the variety of terminal environments:

```
Windows Terminal      -- proper TTY, isTTY = true
VS Code terminal      -- proper TTY, isTTY = true
PowerShell ISE        -- NOT a TTY, isTTY = false
ConEmu                -- usually TTY, isTTY = true
Git Bash (MINGW)      -- varies, may report differently
CI/CD (no terminal)   -- NOT a TTY, isTTY = false
Piped: cmd | node     -- NOT a TTY, isTTY = false
```

Additionally, Windows terminals historically had limited ANSI support. Modern Windows 10/11 and Windows Terminal support ANSI sequences natively, but older environments or certain configurations may not:

```javascript
// More robust color support detection for Windows
function supportsColor() {
  if (!process.stdout.isTTY) return false;
  
  // Windows Terminal, VS Code set this
  if (process.env.COLORTERM === "truecolor" || process.env.COLORTERM === "24bit") {
    return true;
  }
  
  // Check Windows-specific env var
  if (process.env.TERM_PROGRAM === "vscode") return true;
  
  // Windows Terminal
  if (process.env.WT_SESSION) return true;
  
  // Fallback: on Windows, check if ANSI is enabled
  if (process.platform === "win32") {
    // Windows 10 1511+ supports ANSI with VirtualTerminalProcessing enabled
    return process.env.TERM !== undefined || process.env.ANSICON !== undefined;
  }
  
  return true;
}
```

## Piping and stream behavior in Node.js

```javascript
// Reading from stdin (when piped to your tool)
// cat file.txt | node app.js
let input = "";
process.stdin.setEncoding("utf8");
process.stdin.on("data", (chunk) => { input += chunk; });
process.stdin.on("end", () => {
  // Process complete input
  console.log("Received:", input.length, "bytes");
});

// Check if stdin has data (is it a pipe or interactive?)
if (!process.stdin.isTTY) {
  // stdin is being piped -- read it
  process.stdin.resume();
} else {
  // Interactive -- show a prompt
  console.log("Enter input:");
}
```

## Exit codes

Exit codes are how a process communicates success or failure to the parent:

```javascript
// Success
process.exit(0);

// Failure
process.exit(1);

// Node.js specific -- exit with error from uncaught exception
process.exitCode = 1;  // set but don't exit yet
```

```powershell
# Check exit code in PowerShell
node app.js
Write-Host "Exit code: $LASTEXITCODE"

# Use in scripts
node app.js
if ($LASTEXITCODE -ne 0) {
  Write-Error "Tool failed with exit code $LASTEXITCODE"
  exit 1
}
```

```batch
:: Check exit code in CMD
node app.js
if %ERRORLEVEL% neq 0 (
  echo Tool failed
  exit /b 1
)
```

**Common exit code conventions:**
- `0` -- success
- `1` -- general error
- `2` -- misuse of shell command (argument error)
- `127` -- command not found (Unix convention, sometimes used)
- `130` -- terminated by Ctrl+C (Unix convention)

## The \r\n problem in terminal output

On Windows, `os.EOL` is `\r\n`. Using it for terminal output (not file I/O) causes rendering bugs:

```javascript
// WRONG for terminal output
process.stdout.write(`Result: ${value}${os.EOL}`);
// The \r moves cursor to column 0 before \n advances the line
// ANSI color codes written after this get applied from column 0
// Text and colors appear to overwrite each other

// CORRECT -- always use \n for terminal output
process.stdout.write(`Result: ${value}\n`);
```

This was an actual Claude Code bug on Windows. Terminal emulators on all platforms handle `\n` correctly. `os.EOL` is for file content, not terminal streams.

```powershell
# Create a script that writes to both

"

# Redirect only stdout
node -e "
process.stdout.write('stdout message\n');
process.stderr.write('stderr message\n');
" > stdout-only.txt
# stdout-only.txt should have only 'stdout message'
# 'stderr message' should appear in terminal
```

```powershell
node -e "process.exit(0)"; Write-Host "Exit: $LASTEXITCODE"  # 0

```

## What to Notice

- `process.stdout.isTTY` is undefined (falsy) when output is piped -- check for `=== true` not just truthiness
- Always write errors to `process.stderr`, not `process.stdout` -- pipes only capture stdout
- `\n` not `os.EOL` for terminal output -- `os.EOL` is for file content
- On Windows, some environments (PowerShell ISE, older ConEmu) misreport TTY status -- the COLORTERM and WT_SESSION env vars are more reliable indicators

- What TTY detection is and why it affects color output and prompts
- Why `os.EOL` should not be used for terminal output on Windows

Strong answer:

"stdout and stderr are separate streams -- tools that mix them break piping because downstream tools receive error messages mixed into their expected data. TTY detection tells the tool whether it is running interactively or piped, which determines whether to output ANSI colors, show progress spinners, or use plain text. On Windows, isTTY is not always reliable -- Windows Terminal sets WT_SESSION, VS Code sets TERM_PROGRAM, so checking those env vars gives a more accurate picture. And os.EOL should never be used for terminal output -- the \\r in \\r\\n causes cursor position issues that garble ANSI escape sequences."
