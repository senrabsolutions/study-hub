# Bun vs Node.js for CLI Tools

## Why this matters

Claude Code's documentation references Bun as a supported runtime alongside Node.js. Anthropic is actively interested in Bun for CLI tooling -- faster startup, native TypeScript support, and a different subprocess model all have direct implications for a tool used by developers all day long.

If an interviewer asks "what do you know about Bun?" and you say "not much," that's a missed opportunity. This lesson closes that gap.

---

## Key Concepts

- Bun is a JavaScript runtime, bundler, package manager, and test runner in a single binary
- For CLI tools, the three most relevant differences are startup time, TypeScript support, and subprocess behavior
- Bun's Windows support exists but is less mature than Node's -- understanding the gaps is directly relevant to this role
- Bun and Node are largely API-compatible but not identical -- the differences matter in specific situations

---

## What Bun is

Bun is built on JavaScriptCore (Safari's JS engine) rather than V8 (Chrome/Node's engine). It ships as a single self-contained binary that replaces multiple tools:

| Tool | Node ecosystem | Bun equivalent |
|---|---|---|
| Runtime | `node` | `bun` |
| Package manager | `npm` / `pnpm` | `bun install` |
| Bundler | `esbuild` / `webpack` | `bun build` |
| Test runner | `jest` / `vitest` | `bun test` |
| Script runner | `npm run` | `bun run` |
| TypeScript runner | `ts-node` / `tsx` | `bun` (native) |

The single-binary design matters for CLI distribution -- instead of requiring Node + npm + ts-node as prerequisites, a Bun-based CLI can potentially ship as one standalone executable.

---

## The startup time difference

This is the most immediately noticeable difference for CLI tools.

Node.js has to spin up the V8 engine, load the module graph, and JIT-compile code before your first line runs. For a web server that runs for hours, this is irrelevant. For a CLI tool that a developer runs dozens of times per day, it adds up.

```bash
# Typical startup comparison for a simple CLI
node dist/index.js      # ~80-120ms cold start
bun src/index.ts        # ~10-30ms cold start
```

For Claude Code specifically, this matters because developers run it in tight loops -- asking questions, getting answers, running commands, repeating. Every 100ms of startup latency compounds across a workday.

**The Windows caveat:** Bun's startup advantage on Windows is smaller than on Linux/macOS because Windows process creation overhead is higher regardless of runtime. The gap narrows but doesn't disappear.

## Native TypeScript support

Node requires a compilation or transpilation step to run TypeScript:

```bash
# Option 1: compile first
npx tsc && node dist/index.js

# Option 2: use ts-node (slow, adds dependency)
npx ts-node src/index.ts

# Option 3: use tsx (faster ts-node alternative)
npx tsx src/index.ts
```

Bun runs TypeScript directly with no extra tooling:

```bash
bun src/index.ts
```

For development workflow this is significant -- no build step means faster iteration. For production distribution it matters less because you'd typically bundle anyway.

**Important nuance:** Bun does not perform type checking. It strips TypeScript types and runs the JavaScript. You still need `tsc --noEmit` in your CI pipeline to catch type errors -- Bun just doesn't surface them at runtime.

## Subprocess behavior differences

This is the most Windows-relevant difference.

Node's `child_process` module has been battle-tested on Windows for over a decade. The quirks are well-documented:

```typescript
// Node -- known Windows behavior
import { spawn } from "child_process";

// On Windows, shell: true uses cmd.exe
// On Unix, shell: true uses /bin/sh
spawn("git", ["status"], { shell: true });

// To be explicit about shell on Windows:
spawn("cmd.exe", ["/c", "git status"]);
```

Bun's `Bun.spawn` API is newer and cleaner:

```typescript
// Bun
const proc = Bun.spawn(["git", "status"], {
  stdout: "pipe",
  stderr: "pipe",
});
const output = await new Response(proc.stdout).text();
```

**The Windows gap:** As of early 2026, Bun's subprocess handling on Windows has more edge cases than Node's. Specifically:

- Pipe behavior with large output buffers differs
- Some signal handling (SIGTERM, SIGKILL) works differently on Windows under Bun
- The `shell` option behavior is less predictable across CMD/PowerShell/Git Bash

For a tool like Claude Code that spawns many subprocesses (running user commands, invoking tools, executing code), these differences require careful testing on Windows specifically.

## Package manager differences

`bun install` is dramatically faster than `npm install` -- typically 10-20x for large dependency trees. For a CLI tool with many dependencies, this affects contributor experience and CI speed.

```bash
# Installing Claude Code dependencies
npm install     # ~45 seconds on a cold cache
bun install     # ~4 seconds on a cold cache
```

**The lockfile difference:** Bun uses `bun.lockb` (a binary lockfile), not `package-lock.json`. If a project switches to Bun for package management, contributors need Bun installed to get deterministic installs. This is a team decision, not just a tooling preference.

---

## API compatibility -- what works, what doesn't

Bun implements most of Node's built-in APIs, but not all. For CLI tools, the relevant gaps:

**Works fine in Bun:**
- `fs`, `path`, `os` -- full compatibility
- `process.argv`, `process.env`, `process.exit` -- full compatibility
- `readline` -- mostly compatible
- `https`, `http` -- mostly compatible
- `child_process.spawn`, `execSync` -- mostly compatible

**Watch for differences:**
- `child_process.fork()` -- Bun implements this but behavior differs
- Native modules (`.node` files) -- limited support, improving
- Some `stream` APIs -- minor behavioral differences
- `worker_threads` -- support exists but less mature

**The practical implication:** A CLI tool written for Node will likely run on Bun with little or no modification. The issues surface in edge cases -- particularly around subprocess management, native modules, and stream handling.

## Quick Lab

### Step 1 -- Install Bun on Windows

```powershell
# Using PowerShell
irm bun.sh/install.ps1 | iex

# Verify
bun --version
```

### Step 2 -- Compare startup time

```powershell
# Create a minimal script
echo "console.log('hello')" > test.js

# Time Node
Measure-Command { node test.js }

# Time Bun
Measure-Command { bun test.js }
```

### Step 3 -- Run TypeScript directly with Bun

```typescript
// hello.ts
const greeting: string = "Hello from TypeScript";
console.log(greeting);
```

```powershell
# No compilation needed
bun hello.ts
```

### Step 4 -- Check what Bun version Claude Code expects

```powershell
# In the Claude Code repo
cat package.json | grep -i bun
cat .bunversion 2>$null
```

## What to Notice

- Bun's startup advantage is real but smaller on Windows than on Unix
- TypeScript support is a developer experience win but doesn't replace `tsc` for type checking
- Subprocess behavior is where Windows-specific Bun bugs are most likely to appear
- The lockfile format change (`bun.lockb`) is a team-wide decision, not just personal preference
