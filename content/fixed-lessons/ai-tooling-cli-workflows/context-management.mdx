# Context Management and Session State

## Why this matters

Context management is one of the hardest unsolved engineering problems in
agentic AI tools. It is also one of the areas where Windows-specific behavior
makes things meaningfully harder -- verbose Windows error messages, longer path
strings, and wordier PowerShell output all consume context faster than their
Unix equivalents.

Understanding how Claude Code manages context, what happens when it runs out,
and how to engineer for context efficiency is directly relevant to building a
reliable Windows experience.

---

## Key Concepts

- The context window is finite -- every token of conversation history, file content, and tool output counts against it
- Claude Code is responsible for managing what goes into context -- the model only sees what Claude Code sends
- Long sessions, large files, and verbose tool output are the three main sources of context pressure
- When context is exhausted, the session fails -- the user experience is a hard stop mid-task
- Good context management is invisible to the user; bad context management causes mysterious failures

---

## What fills up the context window

Every token sent to the model in a request counts against the context limit.
In a Claude Code session, the context accumulates from:

```
System prompt              ~2,000 - 10,000 tokens (fixed per session)
Conversation history       grows with every exchange
Tool call requests         small -- JSON schemas and inputs
Tool results               CAN BE VERY LARGE -- file contents, command output
Error messages             Windows errors are often verbose
```

The tool results are the main culprit. Reading a large file, running a command
that produces extensive output, or accumulating many tool call cycles across a
long session all eat context rapidly.

---

## Windows-specific context pressure

Windows generates more tokens than Unix equivalents in several common situations:

### Path strings

```
Unix:    /usr/local/bin/node                      ← 22 chars
Windows: C:\Users\logan\AppData\Roaming\npm\node  ← 42 chars
```

Every file path in a tool result is nearly twice as long on Windows. In a
session that reads many files, this adds up.

### Error messages

```
Unix error:
  ENOENT: no such file or directory, open 'config.json'

Windows error:
  The system cannot find the file specified.
  Error Code: 2
  Error opening file: C:\Users\logan\project\config.json
  Please verify the file exists and you have read permissions.
```

Windows error messages are often 3-5x more verbose than their Unix counterparts.
Every error that gets included in a tool result consumes more context.

### PowerShell output

```bash
# Unix command output
git status
# M src/index.ts
# ?? new-file.ts

# PowerShell equivalent output
git status
#         Modified:   src/index.ts
#
# Untracked files:
#   (use "git add <file>..." to include in what will be committed)
#         new-file.ts
```

PowerShell and many Windows CLI tools produce formatted output with more
whitespace, headers, and decoration than Unix equivalents. Each extra character
is a token Claude Code doesn't need to send but might.

## Context management strategies

### Strategy 1: Truncate large tool results

Rather than sending an entire large file to the model, send a summary or the
most relevant portion:

```typescript
async function readFile(path: string, options?: { maxTokens?: number }): Promise<ToolResult> {
  const content = await fs.readFile(path, "utf8");
  const normalized = content.replace(/\r\n/g, "\n");

  const maxChars = (options?.maxTokens ?? 4000) * 4; // ~4 chars per token

  if (normalized.length > maxChars) {
    const truncated = normalized.slice(0, maxChars);
    return {
      content: truncated,
      truncated: true,
      originalLength: normalized.length,
      message: `File truncated to ${maxChars} characters. Full file is ${normalized.length} characters.`
    };
  }

  return { content: normalized };
}
```

### Strategy 2: Strip unnecessary output from commands

Windows command output often includes headers, decorators, and progress
indicators that aren't useful to the model:

```typescript
function cleanCommandOutput(output: string): string {
  return output
    .replace(/\r\n/g, "\n")          // normalize line endings
    .replace(/^\s*\n/gm, "")         // remove blank lines
    .replace(/^PS [A-Z]:\\.*?> /gm, "") // strip PowerShell prompt echoes
    .trim();
}
```

### Strategy 3: Summarize conversation history

For long sessions, Claude Code can compress older exchanges into a summary
and drop the detailed history:

```typescript
// Pseudocode -- real implementation uses the API
async function summarizeOldContext(
  messages: Message[],
  keepRecent: number = 10
): Promise<Message[]> {
  if (messages.length <= keepRecent) return messages;

  const toSummarize = messages.slice(0, -keepRecent);
  const recent = messages.slice(-keepRecent);

  // Ask the model to summarize the older portion
  const summary = await getSummary(toSummarize);

  return [
    { role: "user", content: `[Earlier session summary: ${summary}]` },
    ...recent
  ];
}
```

### Strategy 4: Warn before hitting the limit

Rather than failing hard when the context window is exhausted, warn the user
while there is still room to finish the current task:

```typescript
function checkContextPressure(
  currentTokens: number,
  maxTokens: number
): "ok" | "warning" | "critical" {
  const ratio = currentTokens / maxTokens;

  if (ratio > 0.9) return "critical";  // < 10% remaining
  if (ratio > 0.75) return "warning";  // < 25% remaining
  return "ok";
}

// In the main loop:
const pressure = checkContextPressure(estimatedTokens, MODEL_CONTEXT_LIMIT);

if (pressure === "critical") {
  console.warn(
    "⚠ Context window nearly full. Consider starting a new session " +
    "or asking Claude to summarize before continuing."
  );
}
```

## Session state beyond context

Context is not the only state Claude Code manages. A session also has:

**Working directory** -- where commands run and relative paths resolve from.
On Windows, this includes the drive letter and must be tracked correctly
as the user changes directories.

**Environment variables** -- captured at session start. Changes made during
the session (like PATH updates after installing a tool) are not automatically
visible unless Claude Code refreshes them.

**Open files / locks** -- Windows file locking means a file read by one tool
call may be locked when another tool call tries to write it. Claude Code
needs to manage this, not leave it to luck.

**Git state** -- Claude Code often uses git context (current branch, staged
changes) to inform its behavior. On Windows, git output formatting and path
handling have quirks that affect how this state is parsed.

## What to Notice

- Context pressure on Windows builds faster than on macOS due to longer paths and more verbose output
- Tool results -- especially file reads and command output -- are the main source of context consumption
- Truncating and cleaning output before sending it to the model is good engineering, not a shortcut
- Session state (working directory, environment, file locks) requires explicit management on Windows
