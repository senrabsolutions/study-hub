# X.509 Certificates: What CLI Engineers Actually Need to Know

## Why this matters

X.509 certificates show up in CLI engineering in three concrete ways: HTTPS connections failing with certificate errors, code signing requirements for distributing tools on Windows and macOS, and corporate proxies intercepting TLS traffic. Understanding what certificates actually are -- not just that they exist -- means you can diagnose these failures instead of guessing.

---

## Key Concepts

- A certificate binds an identity (domain name, organization) to a public key
- Trust is established through certificate chains -- your system trusts a root CA, the root CA signs intermediate CAs, intermediates sign end-entity certificates
- Certificate stores on Windows and macOS are separate from Node.js's built-in CA bundle
- Code signing on Windows uses certificates to prove a binary came from a verified publisher
- Expiration, revocation, and purpose constraints are the main reasons valid-looking certificates fail

---

## What a certificate actually contains

An X.509 certificate is a structured data file that contains:

```
Subject:        CN=api.anthropic.com, O=Anthropic, C=US
Issuer:         CN=DigiCert TLS RSA SHA256 2020 CA1 (the CA that signed this)
Public Key:     [RSA 2048-bit public key]
Valid From:     2024-01-01
Valid Until:    2025-01-01
Serial Number:  0A:1B:2C:3D:...
Extensions:
  Subject Alternative Names: api.anthropic.com, *.anthropic.com
  Key Usage: Digital Signature, Key Encipherment
  Extended Key Usage: TLS Web Server Authentication
  Basic Constraints: CA:FALSE  (not a CA, cannot sign other certs)
Signature:      [DigiCert's signature over all the above]
```

The signature is the critical part. DigiCert used their private key to sign this certificate. Your system trusts DigiCert's root certificate (it comes pre-installed in your trust store). So when you connect to `api.anthropic.com`, your system:

1. Gets the server's certificate
2. Verifies DigiCert's signature over it
3. Checks DigiCert's certificate is in the trust store
4. Checks the hostname matches Subject Alternative Names
5. Checks the current date is within Valid From/Until
6. If all checks pass: trust established

## Certificate chains

A single root CA cannot sign every certificate -- that would be a security risk. Instead, root CAs sign intermediate CAs, which sign end-entity certificates:

```
DigiCert Root CA G2          (in your trust store, self-signed)
  |
  +-- DigiCert TLS RSA SHA256 2020 CA1   (intermediate, signed by root)
        |
        +-- api.anthropic.com            (end-entity, signed by intermediate)
```

When your browser or CLI tool connects to a server:
1. The server sends its certificate plus any intermediate certificates
2. Your system builds the chain from the end-entity up to a trusted root
3. If the chain reaches a trusted root: valid
4. If not: `UNABLE_TO_VERIFY_LEAF_SIGNATURE` or similar error

**The missing intermediate certificate problem:**

Servers sometimes fail to send the intermediate certificates. The end-entity certificate is valid, but your system cannot build a complete chain to a trusted root. This causes TLS errors that look like the certificate is invalid but it is not -- the server is just misconfigured.

```bash
# Check if a server sends the full chain
openssl s_client -connect api.anthropic.com:443 -showcerts

# Look for multiple -----BEGIN CERTIFICATE----- blocks
# One = only end-entity (may fail if intermediate is missing from trust store)
# Two or more = chain included (correct)
```

## Windows certificate stores

Windows maintains several certificate stores, separate from Node.js:

```powershell
# List trusted root CAs
Get-ChildItem Cert:\LocalMachine\Root | Select-Object Subject, Thumbprint, NotAfter | Sort-Object Subject

# List personal certificates (your own)
Get-ChildItem Cert:\CurrentUser\My

# List intermediate CAs
Get-ChildItem Cert:\LocalMachine\CA

# Find a specific certificate
Get-ChildItem Cert:\LocalMachine\Root | Where-Object { $_.Subject -like "*DigiCert*" }

# Check if a certificate is expired
Get-ChildItem Cert:\LocalMachine\Root | Where-Object { $_.NotAfter -lt (Get-Date) }
```

```batch
:: CMD equivalent
certutil -store Root         :: trusted roots
certutil -store My           :: personal
certutil -store CA           :: intermediate CAs
```

**The Node.js separation problem:**

Node.js bundles its own set of trusted CA certificates and does not use the Windows certificate store by default. This is why a corporate CA certificate installed in Windows does not automatically make Node.js trust connections signed by that CA.

## Code signing on Windows

Windows requires or recommends code signing for:
- Executables downloaded from the internet (Gatekeeper equivalent)
- PowerShell scripts (when execution policy requires signing)
- Driver packages (required)
- Distributing via winget or Chocolatey (expected)

```powershell
# Check if an executable is signed
Get-AuthenticodeSignature "C:\path\to\tool.exe"

# Output shows:
# Status: Valid  (or NotSigned, HashMismatch, etc.)
# SignerCertificate: [certificate info]
# TimeStamperCertificate: [timestamp info]

# Sign a script (requires a code signing certificate)
Set-AuthenticodeSignature -FilePath ".\script.ps1" -Certificate $cert
```

**Why unsigned tools cause problems:**

When a user downloads an unsigned executable, Windows SmartScreen may block it or show a warning. In enterprise environments, unsigned code may be blocked entirely by Group Policy. For Claude Code distribution, code signing is essential for a smooth install experience.

## Reading certificate errors

Common certificate errors and their meaning:

| Error | Meaning |
|---|---|
| `CERT_HAS_EXPIRED` | Certificate's NotAfter date is in the past |
| `UNABLE_TO_VERIFY_LEAF_SIGNATURE` | Cannot build chain to trusted root |
| `SELF_SIGNED_CERT_IN_CHAIN` | A self-signed cert is in the chain (often corporate proxy) |
| `CERT_COMMON_NAME_INVALID` | Hostname does not match Subject Alternative Names |
| `ERR_TLS_CERT_ALTNAME_INVALID` | Same as above (Node.js version) |
| `CERT_REVOKED` | Certificate was explicitly revoked |

```javascript
// In Node.js, TLS errors come through the error event
https.get("https://example.com", (res) => { ... })
  .on("error", (err) => {
    console.error("Code:", err.code);     // e.g., "CERT_HAS_EXPIRED"
    console.error("Message:", err.message);
  });
```

## Quick Lab

### Step 1 -- Inspect a real certificate

```powershell
# View certificates in your trust store
Get-ChildItem Cert:\LocalMachine\Root | 
  Where-Object { $_.Subject -like "*Microsoft*" } |
  Select-Object Subject, NotAfter, Thumbprint |
  Format-List

# Count how many root CAs you trust
(Get-ChildItem Cert:\LocalMachine\Root).Count
```

### Step 2 -- Check a website's certificate chain

```powershell
# Using .NET to inspect a remote certificate
$request = [System.Net.WebRequest]::Create("https://api.anthropic.com")
$response = $request.GetResponse()
$cert = $request.ServicePoint.Certificate
Write-Host "Subject: $($cert.Subject)"
Write-Host "Issuer: $($cert.Issuer)"
Write-Host "Expires: $($cert.GetExpirationDateString())"
$response.Close()
```

### Step 3 -- Check Node.js CA bundle

```javascript
// How many CAs does Node trust?
const tls = require("tls");
console.log(`Node trusts ${tls.rootCertificates.length} root CAs`);
```

## What to Notice

- Certificate stores in Windows and Node.js are separate -- installing a corporate CA in Windows does not affect Node.js trust
- Missing intermediate certificates are a common server misconfiguration that causes valid certificates to fail
- `SELF_SIGNED_CERT_IN_CHAIN` almost always means a corporate TLS-intercepting proxy, not a real certificate problem
- Code signing is not optional for professional CLI tool distribution on Windows
