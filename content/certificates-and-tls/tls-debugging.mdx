# Debugging TLS and Certificate Errors in CLI Tools

## Why this matters

SSL/TLS errors are one of the most frustrating CLI failures for Windows users. They appear as cryptic messages like `UNABLE_TO_VERIFY_LEAF_SIGNATURE` or `SELF_SIGNED_CERT_IN_CHAIN`, and users have no idea what to do.

As a Windows platform engineer for a CLI tool, you will encounter these in enterprise environments where corporate proxies intercept TLS traffic, in development environments with self-signed certs, and in Windows trust store mismatches.

---

## Key Concepts

- TLS errors mean Node.js couldn't verify the server's certificate
- Node.js maintains its **own** certificate store — separate from Windows
- Enterprise environments often use corporate CAs that aren't in Node's store
- You should never disable TLS verification in production

---

## How TLS verification works

When your CLI makes an HTTPS request, Node.js:

1. Connects to the server
2. Receives the server's certificate chain
3. Checks the chain against its **built-in CA bundle** (from the `ca-certificates` package bundled with Node)
4. If the chain doesn't trace back to a trusted root CA → error

The key issue on Windows: **Node does not use the Windows certificate store by default.** A certificate that Windows trusts completely may still cause Node to throw.

---

## Common TLS errors and what they mean

### `UNABLE_TO_VERIFY_LEAF_SIGNATURE`

The server's certificate exists but Node can't verify the signing chain. Usually means an intermediate certificate is missing or the root CA isn't in Node's bundle.

```
Error: unable to verify the first certificate
    at TLSSocket.onConnectEnd (_tls_wrap.js:...)
```

---

### `SELF_SIGNED_CERT_IN_CHAIN`

A self-signed certificate appears somewhere in the chain. Common in:
- Internal development servers
- Corporate proxies that perform TLS inspection

```
Error: self signed certificate in certificate chain
```

---

### `CERT_HAS_EXPIRED`

The certificate's `notAfter` date has passed.

```
Error: certificate has expired
```

---

### `ERR_TLS_CERT_ALTNAME_INVALID`

The hostname doesn't match the certificate's Subject Alternative Names (SANs).

```
Error: Hostname/IP does not match certificate's altnames
```

---

## The corporate proxy problem

This is the most common enterprise Windows issue.

Many corporate environments deploy a **TLS-intercepting proxy**:

```
Your CLI → Corporate Proxy (re-signs with corporate CA) → Internet
```

The proxy decrypts your HTTPS traffic, inspects it, then re-encrypts it with a corporate CA certificate. From Node's perspective, every HTTPS connection now has a certificate signed by an unknown CA — and Node throws.

**How to identify this:**

```powershell
# Check if a proxy is configured
$env:HTTPS_PROXY
$env:HTTP_PROXY
netsh winhttp show proxy
```

**The fix — add the corporate CA to Node:**

```powershell
# Get the corporate CA cert (usually available from IT or extractable from browser)
# Then point Node at it:
$env:NODE_EXTRA_CA_CERTS = "C:\path\to\corporate-ca.crt"
```

Or in your tool's configuration:
```typescript
import { readFileSync } from "fs";
import https from "https";

const agent = new https.Agent({
  ca: readFileSync("C:\\path\\to\\corporate-ca.crt"),
});
```

## Debugging TLS errors step by step

### Step 1 — Get the actual certificate chain

```powershell
# Using openssl (if available via Git Bash or WSL)
openssl s_client -connect api.anthropic.com:443 -showcerts

# Or use PowerShell
$request = [System.Net.WebRequest]::Create("https://api.anthropic.com")
$response = $request.GetResponse()
$cert = $request.ServicePoint.Certificate
Write-Output $cert
```

### Step 2 — Check Node's TLS behavior directly

```javascript
// test-tls.js
const https = require("https");

https.get("https://api.anthropic.com", (res) => {
  console.log("Success:", res.statusCode);
  const cert = res.socket.getPeerCertificate(true);
  console.log("Issuer:", cert.issuer);
  console.log("Subject:", cert.subject);
}).on("error", (err) => {
  console.error("TLS Error:", err.message);
  console.error("Code:", err.code);
});
```

### Step 3 — Check the Windows certificate store

```powershell
# List trusted root CAs in Windows
Get-ChildItem Cert:\LocalMachine\Root | Select-Object Subject, Thumbprint, NotAfter

# Check if a specific cert exists
Get-ChildItem Cert:\LocalMachine\Root | Where-Object { $_.Subject -like "*Corporate*" }
```

### Step 4 — Use NODE_EXTRA_CA_CERTS

```powershell
$env:NODE_EXTRA_CA_CERTS = "C:\certs\corporate-ca.crt"
node test-tls.js
```

---

## What you should never do

```javascript
// NEVER do this — it disables all TLS verification
process.env.NODE_TLS_REJECT_UNAUTHORIZED = "0";

// Or this
const agent = new https.Agent({ rejectUnauthorized: false });
```

This is sometimes suggested online as a "quick fix." It eliminates all TLS protection and should never appear in production code. If you see it in a codebase, flag it.

```powershell
# Run against a self-signed cert test site

```

## What to Notice

- Node's CA bundle is separate from Windows — corporate CAs require explicit configuration
- `NODE_EXTRA_CA_CERTS` is the safe, correct way to add custom CAs
- TLS errors in enterprise environments are almost always a corporate proxy or missing CA
- The error code (e.g., `SELF_SIGNED_CERT_IN_CHAIN`) tells you exactly what's wrong
