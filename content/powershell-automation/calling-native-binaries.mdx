# Calling Native Binaries from PowerShell

## Why this matters

One of the most frustrating real-world issues:

> “This command works in bash… but breaks in PowerShell.”

PowerShell is not just another shell -- it behaves differently when calling native tools.

This leads to:

- broken arguments  
- quoting issues  
- unexpected parsing  
- inconsistent CLI behavior  

---

## Key Concepts

PowerShell runs two types of commands:

### 1. Cmdlets (PowerShell-native)

```powershell
Get-ChildItem
```

- Return structured **objects**
- Designed for PowerShell pipelines

---

### 2. Native Binaries (external tools)

```powershell
git status
node script.js
```

- Return **plain text**
- Use traditional CLI argument parsing

---

## Quick Summary

PowerShell treats native binaries differently than cmdlets. Argument parsing and quoting can behave differently than in other shells, causing subtle bugs.

---

## Real-World Scenario

You run:

```bash
mytool --name="John Doe"
```

Works in bash.

In PowerShell:

```powershell
mytool --name="John Doe"
```

Fails or behaves incorrectly.

---

## Why This Happens

- PowerShell parses arguments before passing them to the tool  
- Quoting rules differ from bash and CMD  
- Native tools expect raw string arguments  

Result:
👉 arguments may be split or modified

---

## What to Notice

- argument arrays may differ between shells  
- quotes are handled differently  
- PowerShell may reinterpret arguments  
- behavior is not always obvious  

---

## Deep Dive (Important)

Execution flow:

```text
User → PowerShell → Native Tool
```

PowerShell **processes arguments first**, then passes them along.

This is different from bash, which passes arguments more directly.

---

## Real-World Example

A CLI tool expects:

```bash
--config="C:\Program Files\app\config.json"
```

In PowerShell:
- path may be split at spaces  
- tool receives incorrect arguments  

Result:
👉 tool fails or behaves unexpectedly
