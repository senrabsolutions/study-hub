# Install Friction on Windows

## Why this matters

The install experience is the first thing a user encounters. On Windows, it is also where the most can go wrong — PATH issues, missing prerequisites, permission errors, antivirus interference, and shell compatibility problems all surface at install time.

A tool that installs cleanly and works immediately earns trust. A tool that requires troubleshooting to install loses users before they've seen what it does. For Claude Code, the Windows install experience is a direct reflection of how much the team cares about Windows developers.

---

## Key Concepts

- Install friction is any step that requires user effort beyond a single command
- Windows adds layers of complexity that Unix tools often ignore
- Prerequisites, PATH, permissions, and shell compatibility are the four main friction points
- Good install design anticipates failure and communicates clearly

---

## The four friction points on Windows

### 1. Prerequisites

Most CLI tools assume certain things are already installed. On macOS, Homebrew handles this. On Windows, there is no universal solution.

**Common prerequisites that cause install failures:**

- Node.js not installed (or wrong version)
- Git not installed
- Visual C++ Redistributables missing (for native modules)
- Windows Build Tools missing (for `node-gyp`)

**Good design:** detect missing prerequisites and tell the user exactly what to install and where.

```
Error: Node.js 18 or later is required.
Download it from: https://nodejs.org
```

**Bad design:** let the install fail with a cryptic error and leave the user to figure it out.

```
npm ERR! gyp ERR! find Python
npm ERR! gyp ERR! find Python Python is not set from command line or npm config
```

---

### 2. PATH

Covered in depth in the Windows Internals track, but worth restating in the DX context: a tool that installs successfully but isn't on PATH feels broken to the user. They typed the install command. It said it worked. And now the command isn't found.

**The user experience of a PATH problem:**

```powershell
npm install -g my-tool
# ✓ added 42 packages in 3s

my-tool --version
# 'my-tool' is not recognized as an internal or external command
```

**What good looks like:** the install command prints clear next steps if PATH may need updating.

```
✓ my-tool installed successfully.

To use it immediately, run:
  $env:PATH += ";$(npm config get prefix)"

Or restart your terminal to pick up the updated PATH.
```

---

### 3. Permissions

Windows has two permission contexts: standard user and administrator. Many tools unnecessarily require admin privileges, which:

- Blocks installation on managed corporate machines
- Creates security concerns for cautious users
- Adds friction (UAC prompts)

**Good design:** install to user-scoped locations that don't require admin.

```powershell
# User-scoped install — no admin required
npm install -g my-tool        # installs to %APPDATA%\npm

# System-scoped install — requires admin
# Avoid this unless there's a strong reason
```

---

### 4. Shell compatibility

A tool installed successfully but only working in one shell is a broken install from the user's perspective.

**The test for a well-installed Windows CLI:**

```powershell
# Should work in all of these:
# PowerShell 5.x (Windows PowerShell)
# PowerShell 7.x (pwsh)
# CMD
# Windows Terminal (any profile)
# Git Bash
my-tool --version
```

If it works in PowerShell but not CMD, or works in Git Bash but not PowerShell, that's an install problem.

## Real-world install failure patterns

### Pattern 1: Works on developer machine, fails for users

The developer has Node 20, Git, and all build tools installed. The user has a fresh Windows machine with none of that. The tool's install instructions don't mention prerequisites.

**Fix:** document and validate prerequisites explicitly.

### Pattern 2: Installs fine, fails silently in CI

CI environments often have different PATH, different Node versions, and may run as a restricted user. A tool that works interactively may fail in automated pipelines because it reads from stdin expecting user input, or assumes an interactive terminal.

**Fix:** detect non-interactive environments and handle them gracefully.

```typescript
const isInteractive = process.stdout.isTTY;
if (!isInteractive && requiresInput) {
  console.error("Error: this command requires interactive mode.");
  process.exit(1);
}
```

### Pattern 3: Antivirus quarantines the executable

Enterprise Windows environments often run aggressive AV scanning. A newly downloaded executable may be quarantined before the user even tries to run it.

**Fix:** code signing. A signed executable is less likely to be flagged.

## What a great install experience looks like

```
$ npm install -g claude

Checking prerequisites...
  ✓ Node.js 20.x detected
  ✓ npm 10.x detected

Installing...
  ✓ claude@1.0.0 installed

Verifying installation...
  ✓ claude found at C:\Users\logan\AppData\Roaming\npm\claude.cmd

Setup complete. Run: claude --help
```

And if something goes wrong:

```
Installing...
  ✓ claude@1.0.0 installed

Verifying installation...
  ✗ claude not found on PATH

To fix this, add the npm bin directory to your PATH:

  PowerShell:
    [Environment]::SetEnvironmentVariable("PATH", $env:PATH + ";$(npm config get prefix)", "User")

  Then restart your terminal.
```

## Quick Lab

### Step 1 — Measure install friction of an existing tool

```powershell
# Time how long a clean install takes and what steps are required
Measure-Command { npm install -g typescript }

# Check what it added to PATH
where.exe tsc
```

### Step 2 — Simulate a PATH problem

```powershell
# Temporarily remove npm bin from PATH
$original = $env:PATH
$env:PATH = ($env:PATH -split ";" | Where-Object { $_ -notlike "*npm*" }) -join ";"

# Try running a global tool
tsc --version  # Should fail

# Restore
$env:PATH = $original
```

### Step 3 — Check if a tool detects non-interactive mode

```powershell
# Pipe stdin to force non-interactive
echo "" | my-tool interactive-command
```

---

## What to Notice

- PATH problems feel like install failures even when the install succeeded
- The gap between "it works on my machine" and "it works for users" is almost always prerequisites and PATH
- Non-interactive detection matters for CI and automation users
- Good install feedback prevents support tickets
