# Windows Process Model Overview

## Why this matters

Every CLI tool, script, and application runs inside a process.

If you understand how processes are created and managed, you can explain and debug:

- why environment variables behave the way they do
- why permissions differ between executions
- how CLI tools launch other tools
- why subprocesses sometimes fail unexpectedly

This is foundational for:
- Node.js CLI tools
- PowerShell automation
- debugging system-level issues

---

## Key Concepts

- A **process** is a running instance of a program
- Each process has:
  - its own memory space
  - its own environment variables
  - a security context (user + permissions)
- Processes are created by other processes (parent → child)
- Child processes inherit:
  - environment variables
  - working directory
  - security context (with some variation)

---

## Quick Summary

All programs run as processes. When one process launches another, the child inherits key properties from the parent, which directly affects behavior.

---

## Quick Lab

### Step 1 — View running processes

```powershell
Get-Process | Select-Object -First 5
```

### Step 2 — Launch a child process

```powershell
Start-Process notepad
```

### Step 3 — Spawn a process from Node

```bash
node -e "require('child_process').exec('notepad')"
```

---

## What to Notice

- Every program you run becomes a process
- Processes can create other processes
- Tools like Node and PowerShell act as parent processes
- Child processes inherit environment and context from the parent

---

## Real-World Example

A Node CLI tool runs a subprocess:

```bash
git clone ...
```

If Git fails, it might not be Git itself — it could be:

- missing PATH in the parent process
- incorrect working directory
- permission issues inherited from the parent

---

## Debugging Strategy

When a subprocess fails:

1. Check what process launched it
2. Inspect environment variables
3. Confirm PATH is correct
4. Check working directory
5. Verify permissions

In PowerShell:

```powershell
Get-Process
```

---

## Interview Angle

You should be able to explain:

- Everything runs inside a process
- Processes are created by other processes
- Child processes inherit environment and context
- Many CLI bugs come from incorrect assumptions about the parent process

Strong answer example:

“When a CLI tool behaves unexpectedly, I think in terms of process boundaries. The tool runs as a process and may spawn child processes. Those children inherit environment variables, working directory, and permissions, so issues often come from what the parent process provided.”
