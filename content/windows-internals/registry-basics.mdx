# Windows Registry Basics for CLI Engineers

## Why this matters

The Windows Registry is where the OS and applications store persistent configuration. For CLI engineers, it matters in three concrete ways: environment variables (including PATH) live there, tool installation state lives there, and application-level configuration that affects CLI behavior lives there.

When a tool installs correctly but does not work, when PATH changes do not stick, when uninstalls leave broken state -- the registry is usually where you look.

---

## Key Concepts

- The registry is a hierarchical key-value database built into Windows
- Top-level containers are called hives -- HKLM for machine-wide, HKCU for the current user
- Environment variables for the system and current user are stored in specific registry keys
- winget, Chocolatey, and MSI installers all read and write the registry
- Registry changes to environment variables require processes to restart to take effect

---

## The hive structure

```
HKEY_LOCAL_MACHINE (HKLM)    -- system-wide, applies to all users
  SOFTWARE\                   -- installed application settings
  SYSTEM\                     -- OS configuration
  SYSTEM\CurrentControlSet\Control\Session Manager\Environment\  -- system PATH and env vars

HKEY_CURRENT_USER (HKCU)     -- current user only
  Environment\                -- user PATH and environment variables
  SOFTWARE\                   -- user-level application settings

HKEY_CLASSES_ROOT (HKCR)     -- file associations and COM registrations
HKEY_USERS (HKU)             -- all user profiles (HKCU is a view of current user here)
HKEY_CURRENT_CONFIG (HKCC)   -- hardware profile for current boot
```

For daily CLI work, you will almost always be in HKLM or HKCU.

## Where environment variables actually live

```powershell
# User PATH (HKCU)
Get-ItemProperty "HKCU:\Environment" -Name "Path"

# System PATH (HKLM)
Get-ItemProperty "HKLM:\SYSTEM\CurrentControlSet\Control\Session Manager\Environment" -Name "Path"

# Read a specific env var from the registry (the authoritative source)
[Environment]::GetEnvironmentVariable("PATH", "User")    # HKCU
[Environment]::GetEnvironmentVariable("PATH", "Machine") # HKLM

# Write to user PATH permanently (survives reboot, no admin needed)
$current = [Environment]::GetEnvironmentVariable("PATH", "User")
[Environment]::SetEnvironmentVariable("PATH", "$current;C:\my\tool\bin", "User")

# Write to machine PATH permanently (requires admin)
$current = [Environment]::GetEnvironmentVariable("PATH", "Machine")
[Environment]::SetEnvironmentVariable("PATH", "$current;C:\my\tool\bin", "Machine")
```

This is what installers do when they add a tool to PATH -- they write to one of these registry keys. When the installation completes and a new terminal opens, the new terminal reads the updated registry and gets the updated PATH.

## Reading the registry from the command line

```powershell
# Navigate like a filesystem
Set-Location HKCU:\Environment
Get-ChildItem
Get-ItemProperty .

# Read a specific value
Get-ItemPropertyValue "HKCU:\Environment" -Name "Path"

# Read all values in a key
Get-Item "HKCU:\Environment" | Select-Object -ExpandProperty Property

# Find all keys containing a search term (useful for debugging installs)
Get-ChildItem HKLM:\SOFTWARE -Recurse -ErrorAction SilentlyContinue |
  Where-Object { $_.Name -like "*nodejs*" }
```

```batch
:: CMD equivalent using reg.exe
reg query "HKCU\Environment" /v Path
reg query "HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Environment" /v Path
```

## Where installers register software

When you install a program via MSI, winget, or Chocolatey, it writes an uninstall entry to the registry. This is how "Add/Remove Programs" and winget know what is installed:

```powershell
# See what is installed (32-bit and 64-bit)
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\*" |
  Select-Object DisplayName, DisplayVersion, Publisher |
  Where-Object { $_.DisplayName } |
  Sort-Object DisplayName

Get-ItemProperty "HKLM:\SOFTWARE\WOW6432Node\Microsoft\Windows\CurrentVersion\Uninstall\*" |
  Select-Object DisplayName, DisplayVersion |
  Where-Object { $_.DisplayName } |
  Sort-Object DisplayName

# Find a specific tool
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\*" |
  Where-Object { $_.DisplayName -like "*Node*" }
```

This is useful when diagnosing "is this actually installed" questions -- sometimes winget shows an install succeeded but the uninstall entry is missing, which means something went wrong.

## Registry value types

Registry values have types -- understanding them helps when reading raw registry output:

| Type | PowerShell | Description |
|---|---|---|
| REG_SZ | String | Plain text -- most common |
| REG_EXPAND_SZ | ExpandString | Text with env var references like `%SystemRoot%` |
| REG_DWORD | DWord | 32-bit integer |
| REG_QWORD | QWord | 64-bit integer |
| REG_BINARY | Binary | Raw bytes |
| REG_MULTI_SZ | MultiString | List of strings |

PATH is stored as REG_EXPAND_SZ because it contains `%SystemRoot%` and other expandable references.

## Debugging installation problems with the registry

**Symptom: Tool installed but not on PATH**

```powershell
# 1. Check what the installer actually wrote to PATH
[Environment]::GetEnvironmentVariable("PATH", "User") -split ";"
[Environment]::GetEnvironmentVariable("PATH", "Machine") -split ";"

# 2. Check if the tool registered itself as installed
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall\*" |
  Where-Object { $_.DisplayName -like "*NodeJS*" -or $_.DisplayName -like "*Node.js*" }

# 3. Check what install location the registry records
Get-ItemProperty "HKLM:\SOFTWARE\OpenJS Foundation\Node.js" -ErrorAction SilentlyContinue
```

**Symptom: Tool shows as installed but behaves oddly**

```powershell
# Find all registry keys related to the tool
Get-ChildItem HKLM:\SOFTWARE -Recurse -ErrorAction SilentlyContinue |
  Where-Object { $_.Name -match "node|nodejs" } |
  Select-Object Name
```

## What to Notice

- `$env:PATH` in a running session and the registry PATH may differ -- the session value was inherited at startup, the registry is the persistent source
- User PATH (HKCU) does not require admin to modify -- machine PATH (HKLM) does
- Installers that write to machine PATH require elevation -- this is why some installs prompt for UAC
- The registry uninstall keys are how Windows knows what is installed -- a missing entry means Windows does not know the software is there

## Interview Angle

Be ready to explain:

- Where PATH is actually stored on Windows and how it differs from the in-memory value
- The difference between HKCU and HKLM and when each requires elevation
- How to use the registry to diagnose a broken installation
- Why a PATH change does not affect currently running processes

Strong answer:

"Environment variables including PATH are stored in the Windows registry -- user PATH in HKCU\\Environment and system PATH in HKLM\\System\\CurrentControlSet\\Control\\Session Manager\\Environment. When a process starts it reads these and copies them into its environment block. That copy is independent -- registry changes after that point require new processes to see the update. For debugging installs I check the registry uninstall keys to verify the installer completed correctly, and I compare the registry PATH against what I see in running sessions to catch cases where an installer added to PATH but the current terminal predates the install."
