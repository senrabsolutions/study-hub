# PATH Conflicts on Windows

## Why this matters

PATH conflicts are one of the most frustrating Windows debugging problems because the symptom -- a wrong version of a tool running, or a tool failing silently -- looks nothing like the cause. You installed the right version. The installer said it succeeded. But the wrong binary is running.

This comes up constantly in developer environments and is one of the most common reasons Claude Code behaves differently on one Windows machine vs another.

---

## Key Concepts

- Windows searches PATH entries in order -- the first match wins
- Multiple versions of a tool can exist on PATH simultaneously
- PATH entries are accumulated from multiple sources with no deduplication
- The order of entries differs between User PATH and System PATH
- Installers often silently add entries without removing old ones

---

## How PATH conflicts happen

When you run `node`, Windows searches your PATH directories in order and runs the first `node.exe` it finds. If you have Node 18 installed via one method and Node 20 installed via another, whichever directory appears first in PATH wins -- regardless of which version is "current."

Common scenarios that create conflicts:

**Multiple package managers installing the same tool:**
```
C:\Program Files\nodejs\         (installed by official installer)
C:\Users\logan\AppData\Roaming\nvm\versions\node\v20.0.0\  (installed by nvm)
C:\ProgramData\chocolatey\bin\   (installed by Chocolatey)
```

If all three are on PATH, the first one found wins. The other two are silently ignored.

**Old installation left behind:**
```
C:\Python27\          (Python 2.7, never uninstalled)
C:\Python311\         (Python 3.11, current)
```

If `C:\Python27` comes first in PATH, `python` resolves to Python 2.7 even though Python 3.11 is "installed."

**User PATH vs System PATH ordering:**

Windows combines User PATH and System PATH. On most systems, User PATH is prepended to System PATH -- meaning user-level installs take priority over system-level installs. But this depends on whether the installer modified User PATH or System PATH.

## Diagnosing a PATH conflict

### Step 1 -- See what is actually running

```powershell
# Where is the command resolving?
Get-Command node | Select-Object -ExpandProperty Source

# Shows only the FIRST match -- the one that wins
where.exe node
```

### Step 2 -- See ALL matches in PATH order

```powershell
# where.exe shows every match, in PATH priority order
where.exe node
# Output:
# C:\Users\logan\AppData\Roaming\nvm\versions\node\v18.0.0\node.exe
# C:\Program Files\nodejs\node.exe
# The first line wins

where.exe python
where.exe git
```

### Step 3 -- Inspect the full PATH

```powershell
# See PATH split into individual entries
$env:PATH -split ";" | Where-Object { $_ -ne "" }

# Or see User PATH and System PATH separately
[Environment]::GetEnvironmentVariable("PATH", "User") -split ";"
[Environment]::GetEnvironmentVariable("PATH", "Machine") -split ";"
```

### Step 4 -- Check which entry comes first

```powershell
# Find which PATH entries contain a specific tool
$env:PATH -split ";" | Where-Object { Test-Path (Join-Path $_ "node.exe") }
```

## Real-world example: the wrong Git

A developer installed Git via the official installer, then later installed GitHub Desktop which ships its own bundled Git. Their PATH now contains:

```
C:\Users\logan\AppData\Local\GitHubDesktop\app-3.3.0\resources\app\git\cmd
C:\Program Files\Git\cmd
```

GitHub Desktop's Git appears first. When they run `git --version` they get an older version. When Claude Code spawns `git` as a subprocess, it gets GitHub Desktop's Git, not the system Git -- which may behave differently, especially for large repos or certain operations.

**Diagnosis:**
```powershell
where.exe git
# C:\Users\logan\AppData\Local\GitHubDesktop\app-3.3.0\resources\app\git\cmd\git.exe
# C:\Program Files\Git\cmd\git.exe
```

**Fix options:**
1. Reorder PATH so the desired entry comes first
2. Uninstall the conflicting version
3. Use full paths in scripts instead of relying on PATH resolution

---

## Fixing PATH order

```powershell
# Open environment variables GUI
rundll32 sysdm.cpl,EditEnvironmentVariables

# Or edit directly via PowerShell (current session only)
$env:PATH = "C:\Program Files\Git\cmd;" + $env:PATH

# Permanently edit User PATH
$current = [Environment]::GetEnvironmentVariable("PATH", "User")
[Environment]::SetEnvironmentVariable("PATH", "C:\Program Files\Git\cmd;$current", "User")
```

## PATH conflicts and Claude Code

When Claude Code spawns subprocesses -- running tests, executing git commands, invoking build tools -- it inherits the PATH of the shell that launched it. If that shell has a PATH conflict, Claude Code will run the wrong version of tools.

This is why "works on my machine" problems are common with Claude Code on Windows. Two developers with the same tools installed but different PATH orders will see different behavior from the same Claude Code commands.

The diagnostic approach is always the same: `where.exe toolname` to see what actually runs, then trace back to which PATH entry is winning.

## Quick Lab

### Step 1 -- Find conflicts on your system

```powershell
# Check if any tool has multiple versions on PATH
foreach ($tool in @("node", "python", "git", "npm")) {
    $locations = where.exe $tool 2>$null
    if ($locations -is [array] -and $locations.Count -gt 1) {
        Write-Host "CONFLICT: $tool found in multiple locations:"
        $locations | ForEach-Object { Write-Host "  $_" }
    } else {
        Write-Host "OK: $tool -> $locations"
    }
}
```

### Step 2 -- Inspect your full PATH for duplicates

```powershell
$entries = $env:PATH -split ";" | Where-Object { $_ -ne "" }
$duplicates = $entries | Group-Object | Where-Object { $_.Count -gt 1 }
if ($duplicates) {
    Write-Host "Duplicate PATH entries:"
    $duplicates | ForEach-Object { Write-Host "  $($_.Name)" }
} else {
    Write-Host "No duplicate PATH entries found"
}
```

### Step 3 -- Verify the version Claude Code would use

```powershell
# Simulate what a subprocess would see
$env:PATH -split ";" | ForEach-Object {
    $node = Join-Path $_ "node.exe"
    if (Test-Path $node) {
        Write-Host "First node.exe found: $node"
        & $node --version
        break
    }
}
```

---

## What to Notice

- `where.exe` always shows results in PATH priority order -- the first result is what runs
- User PATH is typically prepended to System PATH, so user installs take priority
- Installers rarely clean up old PATH entries when updating -- conflicts accumulate over time
- The PATH a subprocess sees is inherited from its parent -- fixing PATH in one shell does not fix subprocesses spawned from other shells
