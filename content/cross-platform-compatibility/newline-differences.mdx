# Newline Differences: CRLF vs LF

## Why this matters

Newline differences are one of those problems that seems trivial until it breaks something important. A shell script with Windows line endings silently fails to execute. A diff shows every line changed when only one was. A tool parses output incorrectly because it does not expect `\r`. Claude Code's terminal rendering had a bug caused by exactly this -- `\r\n` in output strings was causing bold text to shift and colors to bleed.

Understanding this deeply means you can diagnose it instantly when it appears.

---

## Key Concepts

- Windows uses CRLF (`\r\n`) -- Carriage Return + Line Feed
- Unix/Linux/macOS uses LF (`\n`) -- Line Feed only
- The `\r` character moves the cursor to the start of the line without advancing
- Git has built-in handling for line endings that can silently transform files
- The difference is invisible in most editors but causes real failures in shells and parsers

---

## Why two characters instead of one?

This comes from typewriters and teletype machines. A carriage return moved the print head back to the start of the line. A line feed advanced the paper one line. Early printers needed both. Unix simplified to just LF. Windows kept CRLF for backwards compatibility with DOS and early hardware.

The practical result: every text file created on Windows ends each line with two bytes (`0x0D 0x0A`) instead of one (`0x0A`).

---

## How it breaks things

### Shell scripts fail silently

```bash
#!/bin/bash
echo "hello"
```

If this script has CRLF line endings and you run it in bash (Git Bash, WSL, Linux):

```
bash: ./script.sh: /bin/bash^M: bad interpreter: No such file or directory
```

The `^M` is the `\r` character. Bash reads the shebang line as `/bin/bash\r` and cannot find that interpreter.

### Terminal output gets garbled

```typescript
// In a Node.js CLI tool:
process.stdout.write(`Result: ${value}${os.EOL}`);
// On Windows, os.EOL = "\r\n"
// The \r moves cursor to start of line
// The next character overwrites from the beginning
// This is exactly the bug Claude Code had
```

### Diffs show every line changed

```
-Hello, world\r
+Hello, world
```

When a file is converted from CRLF to LF or vice versa, every single line shows as modified in git diff. This makes code review impossible and `git blame` useless.

### String parsing breaks

```javascript
const lines = fileContent.split("\n");
// lines[0] = "first line\r"  <-- \r is still there
lines[0].trim() === "first line"  // true
lines[0] === "first line"         // false -- \r at end
```

This is a subtle bug -- the line looks correct when printed but comparisons fail because the `\r` is there but invisible.

## Detecting line endings

```powershell
# See raw bytes -- look for 0D 0A (CRLF) vs 0A alone (LF)
Format-Hex .\file.txt | Select-Object -First 5

# Check with PowerShell
$content = Get-Content .\file.txt -Raw
if ($content -match "\r\n") {
    Write-Host "CRLF (Windows)"
} elseif ($content -match "\r") {
    Write-Host "CR only (old Mac)"
} else {
    Write-Host "LF (Unix)"
}
```

```bash
# In bash / Git Bash
file script.sh
# script.sh: ASCII text, with CRLF line terminators  <-- has Windows endings

# See with cat
cat -A file.txt
# Lines with CRLF end with ^M$
# Lines with LF end with $
```

## Git and line endings

Git has built-in line ending handling that silently converts files on checkout and commit. This is controlled by `core.autocrlf`:

```bash
# Check current setting
git config --global core.autocrlf

# Common values:
# true    -- checkout CRLF, commit LF  (Windows developer default)
# input   -- checkout as-is, commit LF  (Unix developer default)
# false   -- no conversion at all
```

**The problem with `autocrlf=true`:** Git converts LF to CRLF on checkout. Shell scripts that need LF endings get CRLF. They then fail when run in Git Bash or WSL.

**The better approach -- `.gitattributes`:**

```
# .gitattributes -- explicit per-file-type rules
* text=auto                    # auto-detect and normalize
*.sh text eol=lf              # always LF for shell scripts
*.ps1 text eol=crlf           # always CRLF for PowerShell scripts
*.md text eol=lf              # always LF for markdown
*.json text eol=lf            # always LF for JSON
*.ts text eol=lf              # always LF for TypeScript
*.tsx text eol=lf             # always LF for TSX
```

This is the correct solution for a cross-platform project. It overrides `autocrlf` on a per-file-type basis.

## Fixing line endings

### Fix a file in PowerShell

```powershell
# Convert CRLF to LF
$content = Get-Content .\file.sh -Raw
$content -replace "`r`n", "`n" | Set-Content .\file.sh -NoNewline

# Convert LF to CRLF
$content = Get-Content .\file.sh -Raw
$content -replace "(?<!\r)`n", "`r`n" | Set-Content .\file.sh -NoNewline
```

### Fix with git

```bash
# After adding .gitattributes, re-normalize all files
git add --renormalize .
git commit -m "Normalize line endings"
```

### Fix in VS Code

Bottom-right corner shows `CRLF` or `LF`. Click it to change for the current file. Or set globally:

```json
// .vscode/settings.json
{
  "files.eol": "\n"
}
```

## Line endings in CLI tool output

For terminal output specifically, always use `\n` regardless of platform. `os.EOL` is for file I/O, not terminal rendering:

```typescript
// WRONG for terminal output on Windows
process.stdout.write(`Done: ${result}${os.EOL}`);

// CORRECT -- always use \n for terminal output
process.stdout.write(`Done: ${result}\n`);
```

The reason: terminal emulators on all platforms (Windows Terminal, VS Code terminal, ConEmu) handle `\n` correctly and advance the cursor. `\r\n` causes the `\r` to move the cursor back to column 0 before the `\n` advances -- which garbles ANSI escape sequences and color codes.

## Quick Lab

### Step 1 -- Detect line endings in a file

```powershell
# Create test files
"line1`r`nline2`r`nline3" | Set-Content -NoNewline .\test-crlf.txt
"line1`nline2`nline3" | Set-Content -NoNewline .\test-lf.txt

# Detect
foreach ($file in @("test-crlf.txt", "test-lf.txt")) {
    $bytes = [System.IO.File]::ReadAllBytes($file)
    $hasCRLF = $false
    for ($i = 0; $i -lt $bytes.Length - 1; $i++) {
        if ($bytes[$i] -eq 13 -and $bytes[$i+1] -eq 10) {
            $hasCRLF = $true; break
        }
    }
    Write-Host "${file}: $(if ($hasCRLF) { 'CRLF' } else { 'LF' })"
}
```

### Step 2 -- See the \r character cause problems

```powershell
# This demonstrates the \r cursor reset behavior
Write-Host "This text`rOVERWRITTEN"
# The \r moves cursor to start, OVERWRITTEN replaces the beginning
```

### Step 3 -- Check your git config

```bash
git config --global core.autocrlf
git config --list | grep -i crlf
```

---

## What to Notice

- The `\r` character is invisible in most editors but causes real failures
- `os.EOL` on Windows returns `\r\n` -- using it for terminal output causes rendering bugs
- `.gitattributes` is the correct long-term solution for mixed-platform teams
- `core.autocrlf=true` solves some problems but creates others -- `.gitattributes` is more precise
