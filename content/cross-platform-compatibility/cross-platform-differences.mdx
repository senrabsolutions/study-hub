# Cross-Platform Differences (Windows vs Linux/macOS)

## Why this matters

A very common real-world issue:

> “This script works on Mac… but breaks on Windows.”

CLI tools often work perfectly on one platform and fail on another.

Common issues include:

- file paths breaking
- commands not being found
- scripts failing due to syntax differences
- newline inconsistencies

If you don’t account for platform differences, your tool will be unreliable.

---

## Key Concepts

### 1. File Paths

- Windows uses:
```
C:\Users\name\file.txt
```

- Linux/macOS use:
```
/home/name/file.txt
```

- Path separators differ:
  - Windows → `\`
  - Unix → `/`

---

### 2. Shell Differences

- Windows:
  - CMD
  - PowerShell

- Unix:
  - bash
  - zsh

Each shell has:
- different syntax
- different quoting rules
- different built-in commands

---

### 3. Environment Variables

- Windows:
```bat
echo %PATH%
```

```powershell
$env:PATH
```

- Unix:
```bash
echo $PATH
```

---

### 4. Newline Differences

- Windows → CRLF (`\r\n`)
- Unix → LF (`\n`)

This can break:
- scripts
- parsing
- version control

---

### 5. Executables

- Windows uses extensions:
  - `.exe`, `.bat`, `.cmd`

- Unix uses:
  - executable permissions (`chmod +x`)

---

## Quick Summary

Different platforms handle paths, shells, environment variables, and execution differently. CLI tools must account for these differences to work reliably.

---

## Real-World Scenario

A script includes:

```bash
rm -rf dist
```

Works on macOS/Linux.

Fails on Windows with:

```
'rm' is not recognized as an internal or external command
```

---

## Why This Happens

- `rm` is a Unix command, not native to Windows  
- the script assumes a Unix shell environment  
- Windows shells (CMD/PowerShell) do not support the same commands  

Result:
👉 the script is not portable

## What to Notice

- Hardcoded paths lead to bugs  
- Shell-specific commands may not work everywhere  
- Environment variable syntax differs  
- Cross-platform tools must normalize behavior  
- Assumptions about the OS are a major source of failure  

---

## Deep Dive (Important)

Most cross-platform failures come from **implicit assumptions**:

- assuming a Unix shell (`rm`, `grep`, etc.)
- assuming path format (`/` vs `\`)
- assuming environment variable syntax  
- assuming executable behavior  

Good tools avoid assumptions and use abstractions.

---

## Debugging Strategy

When something breaks cross-platform:

### 1. Identify platform differences
- Windows vs Linux/macOS  

---

### 2. Check path handling
- avoid hardcoding  
- use platform-aware libraries  

---

### 3. Check shell-specific commands
- replace with cross-platform alternatives  

---

### 4. Normalize environment handling
- consistent variable usage  

---

### 5. Use platform-aware tools

Examples:
- Node `path` module  
- cross-platform CLI utilities  

---

## Interview Angle
